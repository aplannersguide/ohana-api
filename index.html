☼ Road Map

The Ohani API is a project from the San Mateo County team of Code for America fellows that is aiming to create open access to community social services, with an initial emphasis on food security.

The planned development roadmap of the Ohana API involves the creation of the read/write database wrapped in a web service API. Additionally, the aim is to create demonstration apps showing what's possible with the API.

  1. May 2013: Demonstration app: Develop demonstration front-end application that can query the database.
    View Demonstration Application Prototype.
  2. June 2013: Readable API: Develop Ohana database that returns read queries over HTTP.
    View Project on GitHub.
  3. June 2013: Admin API interface: Develop admin application that has read/write access to update database data.
  4. Later 2013: Community-built innovation/apps: Additional applications, such as an SMS (text messaging) search interface for database could be developed to use the data provided by the Ohana API.

☈ Tackle a Challenge!

► Challenge 1: Keeping data current

Challenge Description:

It is critical that the community services listed in the database are up-to-date. Therefore an effective means of keeping the data current needs to be established. This needs to be an administrative layer—an admin app—built on top of the API that can write changes to the stored data through the Ohana API. The following are three possible modes of keeping the data up-to-date:

  1. The moderator model: All users can flag bad data and submit changes, which need the approval of moderators to be applied. This is a similar model to yelp.com.
  2. The wiki model: All users can edit the data and the admin system would need to keep track of changes and allow rollbacks, etc. of malicious/incorrect edits. This is a similar model to wikipedia.org.
  3. The administrator model: Administrator users can log into an admin application to make changes to the data and they will be solely responsible for checking its validity. This is a similar model to a wordpress blog or similar.

Our preference for the challenge is to create the moderator model; however, we would welcome experimentation with the other two approaches if desired. We believe the moderator model to be the best suited for social services, as data of this sort depends heavily on being accurate, while at the same time it is also used by the community at large.

An idea for a moderator model is providing a means to edit the data or “flag” bad data in the database, which any user could do through a frontend application. A registered “moderator” could log into a separate admin application and accept or reject the flagged/edited changes to the data. Moderators could be awarded some form of “points” for their participation to encourage the moderation of edits submitted by the general public. Points could be, for instance, an aggregate number of edits that a particular moderator has accepted and rejected or similar (Fig. 1).

Functional Specification:

  • Web-based application for viewing database fields with the ability to flag/edit entries.
  • Complementary administrative application for viewing flagged data/edited data submitted by the general public, which can then be accepted or rejected.
  • Admin app should require login credentials and have an authenticated means to perform write operations with the database through the Ohana API.

► Challenge 2: SMS Interface

Challenge Description:

Many people looking for community services may not have easy access to the Internet. They do, however, have easier access to SMS-capable phones. Therefore, an important feature of this application would be the ability to query the Ohana API database using text messaging.

Functional Specification:

  • Create an SMS gateway wrapper around the Ohana API that would allow queries to be sent to the API via text message.
  • Provide SMS-based queries with a response formatted appropriately for a mobile phone, for instance, if a search for “food” was performed, the top names and phone numbers of the nearest locations may be returned under that service category, but no additional information (unless the query was more specific).
